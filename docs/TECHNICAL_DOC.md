# LineGuard — Brief Technical Documentation

## Core idea

Every submission in this track can detect a sharp move. Almost none can act on
one. LineGuard is an autonomous in-play risk-control agent: it holds paper
positions on World Cup 1X2 markets and, the instant the market moves, re-prices
every position with a closed-form hedge, routes action through a seeded
simulated venue, reconciles the actual fills, and anchors decisions on Solana
devnet.

The canonical evaluation covers 29 real World Cup matches, 2,876,085 raw TxLINE
odds rows (pre-dedup), 176,286 deduplicated 1X2 updates, and 328 signals. The
immediate lock mean is -1.56 per 100 stake, and the unhedged terminal mean is
-34.92 per 100 stake. All figures are generated by:

```bash
python -m lineguard.results --data data/ --output results/
```

This is a risk-control result, not a profitability or guaranteed-return claim.
Spikes are ephemeral; human reaction time cannot capture them.

## The mathematics

The decision path is deterministic; no LLM participates in trading decisions.
For a stake S on outcome i entered at decimal odds a, with current odds
(o1,o2,o3), every hedging intent reduces to one closed form:

```text
h_j = T*p_j   (j!=i),   T = (S+F)/q_i,   q_i = 1 - sum_{j!=i} 1/o_j
```

where F is the guaranteed net P/L in every losing state. Setting the win state
equal gives the lock-profit floor:

```text
F_lock = S(a*q_i - 1)
```

The current automated test suite contains 64 automated tests passing, including
brute-force verification of terminal P/L equality across all three outcomes.

## 5+1 validation model

- G1 Freshness
- G2 Demargin consistency
- G3 Price consistency
- G4 Range sanity
- G5 Timestamp monotonicity
- G6 Merkle provenance spot-check against TxLINE's on-chain validation views

G1-G5 validate operational data usability. G6 performs a cryptographic Merkle
spot-check.

G6 does not claim that every incoming packet is fully verified on-chain. It
performs a targeted Merkle provenance spot-check against TxLINE's on-chain
validation views.

Solana is used for provenance verification and decision-audit anchoring, not
sportsbook hedge execution.

## Pipeline

```text
ingest (SSE odds+scores, reconnect with JWT renewal, replay mode for judging)
-> Guard (G1-G5 operational checks plus G6 Merkle provenance spot-check at
   decision boundaries; fail => decision refused, refusal itself anchored)
-> Signal (dual-gate z-score, |z|>=2.5 and |dp|>=0.04, cooldown, plus
   score-event attribution: goal-driven vs true sharp)
-> Risk (closed-form desk: OPEN at kickoff, LOCK at take/stop bands)
-> Execute/Reconcile (seeded simulated venue fills, partial-fill handling,
   realized floor calculation)
-> Anchor (sha256 of each decision as an SPL Memo, agent's own wallet)
```

## TxLINE Endpoints and On-Chain Interfaces Used

Txoracle devnet program ID:
`6pW64gN1s2uqjHkn1unFeEjAwJkPGHoppGvS715wyP2J`.

- `POST /auth/guest/start` obtains the guest JWT used to begin TxLINE access.
- On-chain `subscribe` using the Txoracle devnet program activates the
  subscription path for the signing wallet.
- `POST /api/token/activate` activates the API token from the confirmed
  subscribe transaction.
- `GET /api/fixtures/snapshot?competitionId=&startEpochDay=` initializes the
  active fixture context.
- `GET /api/odds/stream` drives probability movement, rolling z-score, and
  lockable hedge calculations.
- `GET /api/scores/stream` provides score-event attribution.
- `GET /api/odds/updates/{day}/{hour}/{interval}` supports deterministic replay
  and evaluation.
- `GET /api/scores/updates/{day}/{hour}/{interval}` supports deterministic
  replay and evaluation with historical score events.
- `GET /api/odds/validation?messageId=&ts=` provides Merkle data for G6 odds
  provenance spot-checking.
- `GET /api/fixtures/validation?fixtureId=&timestamp=` provides Merkle data for
  G6 fixture provenance spot-checking.
- `validate_odds` via `simulateTransaction` spot-checks odds provenance against
  TxLINE's on-chain validation view.
- `validate_fixture` via `simulateTransaction` spot-checks fixture provenance
  against TxLINE's on-chain validation view.

The odds-validation route uses `ts`, while the fixture-validation route uses
`timestamp`. These parameter names are not interchangeable.

## Execution lifecycle

LineGuard does not merely calculate a theoretical hedge. Every trigger runs a
full simulated-execution lifecycle:

```text
PROPOSED -> SUBMITTED -> FILLED / PARTIALLY_FILLED / REJECTED / CANCELLED
-> RECONCILED
```

The venue simulator uses seeded, deterministic frictions: latency, bounded
slippage, fill probability, per-leg liquidity caps, quote halts, and retry
behavior. Reconciliation recomputes the realized floor from actual fills, so
partial fills and single-leg risk are measured and anchored, not assumed away.

## Current scope and boundaries

Current scope:

- live TxLINE ingestion;
- deterministic autonomous policy logic;
- simulated venue execution;
- execution reconciliation;
- stale-input refusal;
- Solana devnet provenance verification;
- Solana devnet decision-audit anchoring;
- public Streamlit dashboard;
- deterministic local replay.

Boundaries:

- no live-money sportsbook execution;
- simulated venue fills are used in the demo;
- Solana is not the betting venue;
- the current canonical evaluation covers 29 football matches;
- future sport or venue integrations are not part of the current evaluation.

The entry policy (back the favorite at kickoff) is deliberately simple; the
product is the risk layer, not the alpha.
